# Known Unknowns

This document is a working acknowledgement of what this portfolio **does not** claim to model accurately. We choose "domain reliability level B — standards alignment" and explicitly disclaim "level C — operational realism" (see design spec §1.3).

## Scope of the "SDF" name — what this models vs the full concept

"SDF" in the target role is Hyundai Motor Group's **Software-Defined Factory**: a factory integrated as a single software OS, hardware and software decoupled across four layers (Hardware → Control → Data Platform → Application), production processes reconfigurable by software update, with a virtual-factory / digital-twin counterpart (HMGICS).

This portfolio deliberately models only the **observability + standards-integration slice** of that vision — roughly the Control → Data Platform → Application data path: ingest machine telemetry (Sparkplug B), model equipment per ISA-95, compute KPIs per ISO 22400, surface them live. The software-defined *control* core is explicitly **out of scope**:

- Virtualized / software-defined control (virtual PLC); hardware-software decoupling.
- Reconfiguring a line by software update / orchestration ("factory as an OS").
- Digital-twin / virtual-factory simulation.
- Deterministic field networking (TSN, OPC UA FX) and the automotive data-space layer (Catena-X / Manufacturing-X).

This is a scope choice, not an oversight: per design spec §1.3 we target reliability level B (standards alignment) and disclaim operational realism, and §16 (Non-Goals) rejects "운영 사실주의 흉내내기". Building the software-defined-control pillars would be exactly that imitation. The honest claim is *"I model the standards-grounded observability slice of an SDF correctly"* — not *"I built an SDF."*

## Operational realism deliberately unmodeled
- Shift handover data consistency — assumed, not validated against any real plant.
- PLC-vendor-specific OPC UA quirks — simulator abstracts them.
- ICS network segmentation (Purdue model) — single docker network in Phase 1.
- Hot patch / maintenance window policy in 24/7 environments — out of scope.
- Multi-region data sovereignty (GDPR, India DPDP) — migration path only (§14 of spec).

## Intentionally unresolved (migration paths declared)
- TimescaleDB Continuous Aggregate × RLS incompatibility (Timescale issue #5787) — not relevant in Phase 1 (no RLS); flagged for Phase 2 schema-per-tenant decision (ADR-0003).
- Kafka exactly-once semantics — at-least-once + idempotent consumer is sufficient (ADR-0005 reasoning).
- 100+ tenant scaling — Phase 2 ends at 3 tenants by design; migration path to Citus/RLS in ADR-0003.

## Phase 1 limitations to be addressed later
- OEE Availability assumes bucket == planned-busy-time (Phase 3 introduces shift schedules).
- OEE `Performance` (ISO 22400-2 *Effectiveness*) can mathematically exceed 1 when the ideal cycle time is set looser than actual throughput. The UC-002 invariant "all four ratios ∈ [0,1]" therefore holds only under the Phase 1 simulator's calibrated cycle times, not for arbitrary real telemetry. Revisit when real ideal-cycle-time sourcing lands.
- Only 5-minute OEE continuous aggregate is implemented (1h and shift in Phase 3).
- Single implicit tenant — multi-tenancy is Phase 2.
- No authn/authz — JWT lands in Phase 2.
- DLQ topic (`dlq.{tenant}`) not yet implemented — invalid records are logged and dropped (see ingest in Task 18). To be added before Phase 2.

### Contract-codegen caveats (surfaced during Section B implementation)
- **[RESOLVED — Section E] Generated code is now the installable `sdf_contracts` package.** `packages/contracts/codegen/python/` carries a hand-authored `pyproject.toml` + `__init__.py` / `py.typed` scaffold (NOT generated — `make all` never rewrites them, so the drift gate stays green), and the generated modules live under `sdf_contracts/{sparkplug,openapi,kafka}/`. Consumers install it editable (`scripts/init.sh`) and import it directly — `sdf_contracts.openapi.sdf_openapi_models`, `sdf_contracts.kafka.machine_telemetry`, `sdf_contracts.kafka.line_state`, `sdf_contracts.sparkplug.sparkplug_b_pb2` — so the apps consume the generated DTOs instead of hand-writing boundary models (contract-first §2). Docker images use a repo-root build context to COPY+install the package alongside each app. There is still **no aggregate `kafka_payloads` module** (directory input forbids single-file codegen output).
- The Sparkplug `*_pb2.py` stub is generated by protoc 29.3 and embeds a runtime-version assertion for protobuf **5.29.3**. The protobuf *gencode ↔ runtime* versions are coupled: `ingest-python` pins `protobuf>=5.29.3,<6` to match the stub. **Bumping to protobuf 6.x requires regenerating the stub** with a matching protoc (and updating the CI protoc pin) — otherwise `ValidateProtobufRuntimeVersion` fails at import.
- `MachineTelemetry.value` is `number | string` (the JSON Schema allows both). The generated Pydantic boundary DTO coerces leniently — a numeric-looking string may become a `float`. Boundary validation is intentionally lax (ADR-0018); re-narrow per `metric` in the `to_domain` conversion, or, if strict typing becomes necessary, tighten the schema (`oneOf` discriminated on `metric`) and regenerate.

### Spec/ADR errata — reconciled during Section D implementation
- **[RESOLVED — ADR-0026] ISO 22400-2 clause number: ADR-0012 said §5; §6 is correct.** `ADR-0012` cited ISO 22400-2:2014 **§5** (4×) for the OEE / A·P·Q definitions, but `GLOSSARY` (`monitoring` → *OEE* / *A·P·Q*), `DOMAIN-NOTES` (§"ISO 22400 — KPI definitions"), and the `monitoring/domain/oee.py` implementation all cite **§6**. Verified against the published standard's **Contents page** (ISO 22400-2:2014, 1st ed. 2014-01-15): **clause 5 = "Elements used in KPI description"** (input time/quantity elements, pp. 5–11) and **clause 6 = "Description of KPIs"** (the KPI definitions incl. OEE, pp. 12–36); the loss-time OEE variant is the *informative* **Annex B**. So **§6 is correct** and `GLOSSARY`/`DOMAIN-NOTES`/code were right. **Resolved 2026-05-24 by [ADR-0026](ADR/0026-oee-iso22400-clause-correction.md)**, which supersedes ADR-0012 with the §6 citation; ADR-0012 status flipped to superseded. No code impact (`oee.py` + tests already cite §6). Source: ISO 22400-2:2014 Contents, iTeh Standards official preview sample — https://standards.iteh.ai/catalog/standards/sist/8a9efc01-6c74-42a2-ad8f-ec19e84b48f0/iso-22400-2-2014

### Edge↔topology identifier resolution (surfaced during Section E bridge implementation)
- The `machine_telemetry` Kafka payload (ADR-0005 SoT) carries **edge-native string keys** — `lineId` = Sparkplug `edge_node_id` (line slug, e.g. `line-a`) and `machineKey` = compound-metric prefix (e.g. `press`) — **not** topology UUIDs. The bridge is a stateless edge normalizer with no topology/DB access, so resolving slugs to `production_line.id` / `machine.id` (uuid PKs in `infra/timescale/init/002_schema.sql`) is the **ingest** service's job at DB-write time (Task 18, deferred). The earlier `lineId`/`machineId` `format: uuid` schema presumed resolution happened *before* Kafka; corrected to edge-native keys in a `fix(contracts)` commit during Section E. `line_state.schema.json` deliberately keeps `lineId: uuid` because its producer is DB-side (monitoring), not the edge bridge — the two payloads sit on opposite sides of the resolution boundary.
- **[RESOLVED — Section E Task 18] Join key = `machine.sparkplug_node_id`, reassembled by ingest.** The simulator publishes `SDF_LINE_ID=line-a` (docker-compose) with machine keys `press|weld|paint|inspect|pack`; the bridge emits those verbatim as `lineId`/`machineKey`. The seed previously stored `machine.sparkplug_node_id='sdf_default/Line A/press'` (built from the *display name* `Line A`), which the edge slug `line-a` never matched. **Resolution:** the seed (`005_seed.sql`) was corrected so `sparkplug_node_id` is the real Sparkplug address `{group}/{edge_node_id}/{machineKey}` = `'sdf_default/line-a/<type>'` (the display name `production_line.name='Line A'` stays, for the UI only). Ingest reassembles `f"{tenantId}/{lineId}/{machineKey}"` and resolves it against the UNIQUE, indexed `machine.sparkplug_node_id`. This needs no schema change and no contract/bridge change; it corrects a genuine modeling bug (a human display name had leaked into a protocol address). The line slug `line-a` is the single canonical line key shared by simulator env, bridge edge_node_id, and seed.

### OEE & line-state derivation (surfaced during Section E ingest/api)
- **OEE CAGG sums absolute counters — the API OEE read bypasses it (Phase 1).** `cycle_count` / `good_count` are *cumulative monotonic counters* (the simulator publishes the running total each tick), but `line_oee_5m` (`infra/timescale/init/004_continuous_aggregates.sql`) does `SUM(value_num)` over them — treating counters as gauges. Summing absolute readings inflates "produced quantity" by orders of magnitude (≈ Σ of the counter samples, not the per-bucket increase), which would drive `performance` far above 1 and break the OEE endpoint. So `PgOeeReader` (Section E) **reads `machine_telemetry` directly**, computing per-machine counter deltas (`max(value_num) − min(value_num)` over a trailing 5-minute window) summed to the line, then runs the Section D `compute_oee`. The CAGG is left in place but unused by the API; a correct CAGG needs a delta-aware aggregation (per-machine `last(…)−first(…)` or `timescaledb_toolkit.counter_agg`) and is **deferred** to a later infra task. Caveat: the trailing-window `max−min` undercounts by the first sample of the window (~1 of ~300 readings at 1 Hz) and assumes no counter reset within the window.
- **Line state is a Phase-1 heuristic, not modeled downtime.** Phase 1 has no dedicated line-state producer (the edge emits only counters, no `state` metric), so the ingest service derives a coarse state in `domain/line_activity.py`: a machine whose `cycle_count` advanced since the last observation is producing, and a line is `RUNNING` if any of its machines is, else `IDLE`. `DOWN` / `CHANGEOVER` are **not derivable** from counts alone and are deliberately never emitted — synthesising them would be unmodeled fiction. `line_state` rows are written only on a state *change* (a transition log), and the API/WebSocket surface this. A real downtime/changeover signal (operator input, PLC state word) is future work.

### Frontend boundary & live-state (surfaced during the frontend-architecture review, ADR-0028/0029)
- **The live WebSocket frame is not yet a first-class contract schema.** ADR-0029 has the frontend validate each WS frame through generated Zod by reusing the `LineStateSnapshot` component (`packages/contracts/openapi/sdf-api.yaml`). But OpenAPI 3.1 has no WebSocket message type, and the API currently broadcasts an untyped `dict` (`apps/api-python/.../monitoring/adapters/ws.py`). Section F must make the live frame conform to (and the broadcaster validate against) the shared component — and if the live frame carries a transition `reason` (UC-001), add `reason` to the `LineStateSnapshot` component *first* (contract-first, regenerate), then let it flow through the mapper. Until then there is no generated Zod for the WS frame.
- **Cache-as-single-source-of-truth assumes the terminal frame is delivered.** The Section E broadcaster drops frames for slow consumers (bounded queue); with the frontend's `staleTime: Infinity` and no polling while connected (ADR-0029), a dropped *last* frame would leave the cached *Line state* stale until the next change or a disconnect. The heartbeat detects a dead socket, not a dropped frame. Mitigation (periodic reconcile or a bounded `staleTime`) is deferred to Section F.

### Frontend tooling (surfaced during the Section F scaffold review, PR #18)
- **`vite.config.ts` is intentionally untyped-by-Node for now; `@types/node` deferred.** The config uses `__dirname` + `node:path`, but it sits *outside* tsc's `include` (`apps/dashboard-react/tsconfig.json` → `["src","tests"]`) and `vite build` is green because Vite shims `__dirname`/`__filename` and resolves `node:path` at bundle time. A bare `@types/node` devDep would be *wrong* here: `tsconfig.json` has no `types` field, so it would auto-include Node globals (`process`, `Buffer`, …) into the **global scope of `src/` including `domain/`**, where the ESLint purity guards (§2) ban browser IO but not `process`/`Buffer` — a silent FC/IS purity regression (verified: `tsc` accepts `process.env` / `Buffer` in a `src/` file once `@types/node` resolves with no `types` allowlist). The correct fix is a dedicated `tsconfig.node.json` (`types: ["node"]`, `include: ["vite.config.ts"]`) for type-coverage of config files **plus** a root `"types"` allowlist so Node globals never leak into `src/`. Deferred — the risk is latent (no `src/contexts/` domain code exists yet) and the config works today; land the split when the domain layer is scaffolded, or when config tooling grows beyond the single vite file.

### Frontend live-state implementation (surfaced during Section F Task 23)
- **No app-level WebSocket heartbeat — deferred, not forgotten.** ADR-0029 / frontend-rules §5 call for "exponential backoff + jitter **and** an app-level heartbeat." Task 23's `WebSocketLineStateReader` (`apps/dashboard-react/.../monitoring/adapters/websocket-line-state-reader.ts`) implements reconnect-on-drop with exponential backoff + jitter, but **not** a heartbeat: the Phase-1 endpoint (`apps/api-python/.../monitoring/adapters/ws.py`) is send-only and pushes *on change*, so it never reads a client ping, and a healthy *Line* can be silent for minutes — there is no honest heartbeat signal to act on (a "no frame in N s → reconnect" watchdog would false-positive on a steady line). A real heartbeat needs a bidirectional ping/pong (a client→server frame the server echoes) or a server keepalive tick; both are backend changes. Deferred until the WS endpoint gains a bidirectional channel.
- **Dropped-last-frame staleness still open (carried from the live-state review above).** The cache holds the domain type with `staleTime: Infinity` and polls only while disconnected (ADR-0029); a *last* frame dropped by the backend's bounded-queue broadcaster on a healthy socket leaves the cached *Line state* stale until the next change or a disconnect. Task 23 did not add a periodic reconcile (a bounded `staleTime` or a low-frequency background refetch that does not race `setQueryData`). Deferred — bounding this staleness honestly depends on the keepalive above.
- **The WS frame still rides the REST `LineStateSnapshot` Zod (no dedicated WS message schema).** Task 23 validates each frame with the generated `zLineStateSnapshot` and runs it through the same mapper as the REST path (ADR-0029, resolving the frontend half of the live-state-contract item above); the backend broadcaster still emits an untyped `dict` that happens to match. A first-class WS message contract plus server-side validation remains open.
