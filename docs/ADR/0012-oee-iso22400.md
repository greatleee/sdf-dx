# ADR-0012: OEE per ISO 22400

- **Status:** superseded by ADR-0026
- **Superseded by:** [ADR-0026](0026-oee-iso22400-clause-correction.md) — corrects the ISO 22400-2 clause citation from §5 to §6 (externally verified). All formulas and decisions (D-1–D-4) below are unchanged; only the clause number was wrong. Read 0026 for the authoritative form.
- **Date:** 2026-05-23
- **Phase:** 1

## Context

The monitoring BC's headline KPI is OEE (Overall Equipment Effectiveness). The project thesis (spec §1.1) is *honest, standards-based modeling* of an unfamiliar domain — so OEE cannot be a hand-rolled "looks about right" formula. It must follow the published international standard, and where Phase 1 simplifies the standard, the simplification must be explicit and logged, not silently baked in.

The relevant standard is **ISO 22400-2:2014** ("Automation systems and integration — Key performance indicators (KPIs) for manufacturing operations management — Part 2: Definitions and descriptions"), §5, which defines OEE and its three component KPIs. GLOSSARY `monitoring` already pins the formulas (*OEE*, *A / P / Q* entries); this ADR is the decision record those entries cite, and the formulas here are kept **identical** to the GLOSSARY.

Two project constraints shape the Phase 1 form:

1. OEE is a **pure computation** — it consumes counts and times and returns ratios. Under ADR-0004 (FC/IS) it must live as a pure domain function with no IO, no clock read, no DB access (`monitoring/domain/oee.py`).
2. Phase 1 has a single time window: the **5-minute continuous aggregate** (`line_oee_5m` CAGG, GLOSSARY `monitoring` → *Continuous Aggregate*). There is no shift-schedule or planned-downtime tracking yet, so the *Availability* denominator (Planned Busy Time) is not yet a tracked quantity.

## Decision

### D-1. OEE follows ISO 22400-2:2014 §5

```
OEE = Availability × Performance × Quality
```

with the three components defined exactly as in the standard (and verbatim with GLOSSARY `monitoring` → *A / P / Q*):

- **Availability** `= APT / PBT` — Actual Production Time / Planned Busy Time.
- **Performance** `= (Ideal Cycle Time × Produced Quantity) / APT`.
- **Quality** `= Good Quantity / Produced Quantity`.

All three components ∈ `[0, 1]`, and **OEE inherits that range** (`[0, 1]`, being a product of three values each in `[0, 1]`).

### D-2. Phase 1 simplification — bucket *is* the Planned Busy Time

The 5-minute continuous-aggregate bucket is treated **as** the Planned Busy Time: `PBT = bucket length`. With Actual Production Time approximated as the full bucket, **Availability is approximated as `1.0`** for Phase 1. This is a known, deliberate simplification: Phase 1 has no shift schedule or planned-downtime model from which a real PBT could be derived, so the bucket stands in for it and Availability collapses to the multiplicative identity. OEE in Phase 1 is therefore effectively `Performance × Quality`, with the Availability term present in the computation but pinned to `1.0`.

This simplification is **logged in `KNOWN-UNKNOWNS.md`** as an open item to be retired in Phase 3 (D-4 / Migration). The GLOSSARY `monitoring` → *A / P / Q* entry already records it ("Phase 1 simplification: A is approximated as 1.0").

### D-3. Only the 5-minute CAGG exists in Phase 1

`line_oee_5m` (the 5-minute window) is the **only** OEE aggregate in Phase 1. The **1-hour and shift-length windows are Phase 3**, introduced together with the planned-busy-time tracking that makes a real Availability computation possible (D-4).

### D-4. Computation is a pure domain function returning failures as values

Per ADR-0004, the OEE computation is a **pure domain function** in `monitoring/domain/oee.py` — no IO, no clock read, no DB access. It receives counts and times as parameters and returns the ratio(s).

Per ADR-0016 (error-as-value), **degenerate inputs return a named failure value, not a raised exception.** The obvious degeneracy is **zero Produced Quantity**, which makes both *Performance* (divides by Produced Quantity via APT-derived terms) and *Quality* (`Good / Produced`) undefined. The function returns a named sum-type case (e.g., `Undefined(reason=...)` / `NoProduction`) rather than raising `ZeroDivisionError`. Other degeneracies (zero APT, zero PBT) are handled the same way. The caller discriminates the outcome with `match`; the HTTP boundary translates a degenerate-OEE case to the appropriate response in the adapter, never in the domain.

## Consequences

### Positive
- OEE is anchored to a citable international standard (ISO 22400-2:2014 §5), supporting the "honest, standards-based modeling" thesis — an interview reader can check the formula against the standard.
- The Availability ≈ 1.0 simplification is explicit and tracked (KNOWN-UNKNOWNS + GLOSSARY), not hidden — the simplification itself is a portfolio signal of standards-literacy rather than a shortcut.
- Pure-function form (ADR-0004) makes the computation property-testable: ranges (`[0,1]`), monotonicity in Good Quantity, and the zero-production degeneracy are all checkable with Hypothesis, zero mocks.
- Errors-as-values (ADR-0016) means the zero-Produced-Quantity case is part of the return type — the caller cannot forget to handle "no production this bucket".

### Negative / Trade-offs
- Phase 1 OEE is *not* a true OEE — Availability is pinned to 1.0, so the headline number overstates effectiveness whenever real downtime occurred inside a bucket. This is acceptable for a Phase 1 vertical slice and corrected in Phase 3, but a careless reader could over-trust the Phase 1 figure. The KNOWN-UNKNOWNS entry is the mitigation.
- A 5-minute window is short for OEE, which is conventionally reported per shift; the 5-minute CAGG is a live-monitoring granularity, and shift/hour windows arrive in Phase 3.
- Returning a named failure for zero production (rather than, say, `0.0` or `null`) pushes a discrimination burden onto every caller — intentional under ADR-0016, but more verbose than a sentinel.

## Migration Path

Forward: **Phase 3 introduces planned-busy-time tracking** (shift schedules, planned-downtime calendar) and **re-derives Availability properly** as `APT / PBT` from tracked quantities, retiring the `Availability = 1.0` approximation. The same Phase 3 work adds the 1-hour and shift-length OEE windows (D-3). At that point the KNOWN-UNKNOWNS entry for the Availability simplification is resolved, and the GLOSSARY `monitoring` → *A / P / Q* note is updated to drop the "approximated as 1.0" clause.

The pure-function shape (D-4) does not change in the migration — only the inputs become real (a tracked PBT replaces the bucket-as-PBT substitution), so no call-site rework is needed beyond supplying the new PBT argument.
